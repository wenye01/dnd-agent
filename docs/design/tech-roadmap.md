## 1. 技术选型决策

### 1.1 后端技术栈（Go 语言）

| 组件 | 选型 | 理由 |
|------|------|------|
| **Web 框架** | Gin | 轻量高性能，生态成熟，适合构建 REST API 和 WebSocket 服务 |
| **WebSocket** | Gorilla WebSocket | Go 语言最成熟的 WebSocket 库，与 Gin 集成良好 |
| **配置管理** | Viper | 支持多格式配置文件、环境变量、热重载 |
| **日志** | Zap / Zerolog | 高性能结构化日志 |
| **状态存储** | 内存 (运行时) + JSONL (持久化) | 单用户本地部署场景，内存存储性能最优；JSONL 格式便于追加和查看 |
| **LLM 客户端** | 自研适配层 | 统一封装 OpenAI 和 Anthropic API，支持流式响应 |
| **序列化** | encoding/json | JSON 用于可读性场景，标准库足够用 |
| **测试** | testify + mockery | 标准测试框架 + Mock 工具 |

### 1.3 LLM 集成方案

| 项目 | 方案 |
|------|------|
| **支持格式** | OpenAI API 格式（兼容多家）+ Anthropic API 格式 |
| **调用方式** | HTTP REST API，支持流式响应（SSE） |
| **适配层设计** | 统一接口 `LLMProvider`，不同提供商实现各自适配器 |
| **配置方式** | 配置文件指定提供商类型、API Key、Base URL、模型名称 |
| **工具调用** | OpenAI Function Calling / Anthropic Tool Use |

**LLM 适配器接口设计**：

```go
type LLMProvider interface {
    // 发送消息，获取响应
    SendMessage(ctx context.Context, messages []Message, tools []Tool) (*Response, error)

    // 流式发送消息
    StreamMessage(ctx context.Context, messages []Message, tools []Tool) (<-chan StreamChunk, error)
}

type Message struct {
    Role    string // "system", "user", "assistant", "tool"
    Content string
    // 工具调用相关字段
    ToolCallID   string
    ToolCalls    []ToolCall
}

type Tool struct {
    Name        string
    Description string
    InputSchema map[string]interface{} // JSON Schema
}
```

### 1.4 通信协议

#### 1.4.1 WebSocket 通信协议

Web 前端与游戏后端之间使用 WebSocket 进行实时双向通信。

**消息格式**：

```typescript
// 客户端 -> 服务端
interface ClientMessage {
  type: 'user_input' | 'map_action' | 'combat_action' | 'subscribe' | 'management';
  payload: any;
  requestId?: string; // 用于请求-响应匹配
}

// 服务端 -> 客户端
interface ServerMessage {
  type: 'state_update' | 'narration' | 'combat_event' | 'dice_result' | 'error' | 'pong';
  payload: any;
  requestId?: string;
}
```

**消息类型说明**：

| 类型 | 方向 | 说明 |
|------|------|------|
| `user_input` | C→S | 用户自然语言输入 |
| `map_action` | C→S | 地图点击/移动/交互 |
| `combat_action` | C→S | 战斗行动选择 |
| `subscribe` | C→S | 订阅状态更新 |
| `management` | C→S | 管理操作 |
| `state_update` | S→C | 游戏状态更新推送 |
| `narration` | S→C | DM 叙述文本 |
| `combat_event` | S→C | 战斗事件（攻击、伤害等） |
| `dice_result` | S→C | 检定/掷骰结果 |
| `error` | S→C | 错误信息 |

#### 1.4.2 MCP 协议（内部）

Client 模块与 Server 模块之间通过 MCP 格式接口通信（进程内调用，无网络开销）。

**工具定义格式**：

```go
type ToolDefinition struct {
    Name        string                 `json:"name"`
    Description string                 `json:"description"`
    InputSchema map[string]interface{} `json:"inputSchema"` // JSON Schema
}
```

**调用格式**：

```go
type ToolCall struct {
    Name       string                 `json:"name"`
    Arguments  map[string]interface{} `json:"arguments"`
}

type ToolResult struct {
    Success bool        `json:"success"`
    Data    interface{} `json:"data,omitempty"`
    Error   string      `json:"error,omitempty"`
}
```

### 1.5 数据存储方案

采用 **内存 + JSONL 持久化** 的全量方案：

| 数据类型 | 存储方案 | 说明 |
|----------|----------|------|
| **游戏状态** | 内存 + JSONL 快照 | 运行时全部在内存，定期/事件触发时持久化到 JSONL |
| **会话消息** | 内存 + JSONL 追加 | 每条消息追加写入 JSONL 文件，支持流式查看 |
| **存档** | JSON 文件 | 独立快照文件，便于备份、迁移和回滚 |
| **剧本数据** | JSON/YAML 文件 | 只读，按需加载到内存 |
| **地图数据** | JSON 文件 | FVTT 格式或转换后的格式，按需加载 |

**JSONL 文件设计**：

```
data/
├── sessions/
│   └── {session_id}/
│       ├── state.jsonl          # 游戏状态快照（每次保存覆盖）
│       ├── messages.jsonl       # 消息历史（追加写入）
│       └── events.jsonl         # 游戏事件日志（追加写入）
├── saves/
│   ├── save_001.json            # 手动存档（完整状态快照）
│   └── save_002.json
├── scenarios/                   # 剧本数据（只读）
└── maps/                        # 地图数据（只读）
```

**JSONL 消息格式示例**：

```jsonl
{"type":"message","timestamp":"2026-03-16T10:30:00Z","role":"user","content":"我想创建一个精灵法师"}
{"type":"message","timestamp":"2026-03-16T10:30:05Z","role":"assistant","content":"好的，让我们开始创建..."}
{"type":"state_change","timestamp":"2026-03-16T10:31:00Z","path":"party.0.name","oldValue":null,"newValue":"艾拉"}
{"type":"dice_roll","timestamp":"2026-03-16T10:31:05Z","formula":"4d6k3","result":[5,6,4,2],"total":15}
```

**内存数据结构**：


**方案优势**：

1. **简单直观**：无需数据库配置和 ORM 学习
2. **易于调试**：JSONL 文件可直接用文本编辑器查看
3. **便于备份**：复制文件夹即可完成备份
4. **版本控制友好**：可以 git 追踪状态变化
5. **性能优异**：内存操作无 I/O 延迟
6. **可恢复性**：JSONL 追加写入，即使崩溃也只丢失最后几条

## 4. 关键技术决策

### 4.1 决策记录

| 决策ID | 决策 | 选择 | 备选方案 | 理由 |
|--------|------|------|----------|------|
| D-001 | 后端语言 | Go | Rust, Node.js | 性能优秀、并发模型简单、编译快速、学习曲线适中 |
| D-002 | Web 框架 | Gin | Echo, Fiber | 生态最成熟、社区活跃、文档完善 |
| D-003 | 状态存储 | 内存 + JSONL | SQLite, BadgerDB | 单用户场景，内存性能最优；JSONL 便于查看和追加 |
| D-004 | 游戏引擎 | Phaser | PixiJS, Three.js | 2D 游戏成熟方案、文档丰富、社区活跃 |
| D-005 | 状态管理 | Zustand | Redux, MobX | 轻量、TypeScript 友好、适合游戏状态 |
| D-006 | 前端框架 | React | Vue, Svelte | 生态最大、组件丰富、团队熟悉 |
| D-007 | 通信协议 | WebSocket | SSE, Long Polling | 双向实时通信需求，WebSocket 最适合 |
| D-008 | 数据格式 | JSON + msgpack | Protobuf | JSON 可读性好，msgpack 可选用于性能优化 |
| D-009 | 部署方式 | 单进程合并 | 微服务分离 | 单用户场景，合并部署更简单高效 |
| D-010 | 存档格式 | JSON 文件 | 二进制格式 | 可读性好、便于调试、大小可接受 |

### 4.2 决策利弊分析

#### D-001: Go vs Rust vs Node.js

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Go (选择)** | 编译快、并发简单、部署简单（单二进制）、内存占用低 | 泛型支持较弱、错误处理繁琐 |
| Rust | 性能最优、内存安全 | 学习曲线陡峭、编译慢 |
| Node.js | 前后端统一语言、生态丰富 | 性能较差、类型需 TypeScript 补充 |

#### D-003: 内存+JSONL vs SQLite vs BadgerDB

| 方案 | 优点 | 缺点 |
|------|------|------|
| **内存+JSONL (选择)** | 性能最优、简单直观、易于调试、可直接查看文件 | 重启需重新加载、无事务支持 |
| SQLite | 支持事务、结构化查询 | 需要 ORM、配置复杂 |
| BadgerDB | 高性能 KV 存储 | 需要额外依赖、调试困难 |

#### D-004: Phaser vs PixiJS

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Phaser (选择)** | 完整游戏框架、场景管理、输入处理、内置物理 | 包体积较大 |
| PixiJS | 更轻量、渲染性能优 | 需要自己实现游戏逻辑 |

## 5. 技术风险清单

### 5.1 高风险

| 风险ID | 风险描述 | 可能性 | 影响 | 缓解策略 |
|--------|----------|--------|------|----------|
| R-001 | **LLM 幻觉导致规则违反** | 高 | 高 | Server 严格执行所有规则验证；关键操作需要显式工具调用；拒绝非法操作并返回错误信息 |
| R-002 | **上下文窗口限制** | 中 | 高 | 实现上下文压缩策略；渐进式披露减少 token 消耗；优先保留关键信息 |
| R-003 | **D&D 5e 规则复杂性** | 高 | 高 | 分阶段实现规则；优先核心规则；建立规则测试用例库 |

### 5.2 中风险

| 风险ID | 风险描述 | 可能性 | 影响 | 缓解策略 |
|--------|----------|--------|------|----------|
| R-004 | **WebSocket 连接不稳定** | 中 | 中 | 实现断线重连机制；持久化关键状态用于恢复；心跳检测 |
| R-005 | **多 LLM 提供商 API 差异** | 中 | 中 | 设计统一适配层；充分测试各提供商；使用抽象接口 |
| R-006 | **FVTT 格式兼容性** | 低 | 中 | 按官方格式设计；提供格式转换工具；做好数据验证 |
| R-007 | **战斗平衡性难以保证** | 中 | 中 | 参考 DMG 遭遇设计指南；提供难度调整选项；收集反馈迭代 |

### 5.3 低风险

| 风险ID | 风险描述 | 可能性 | 影响 | 缓解策略 |
|--------|----------|--------|------|----------|
| R-008 | **剧本解析工作量大** | 高 | 低 | 当前版本使用手动预处理；解析工具后续开发 |
| R-009 | **前端性能问题** | 低 | 中 | 使用 Phaser 的优化渲染；避免过度重绘；按需加载资源 |
| R-010 | **JSONL 文件损坏** | 低 | 中 | 追加写入原子性；定期备份；状态快照恢复 |
