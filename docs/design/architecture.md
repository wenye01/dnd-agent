## 1. 架构目标和原则

### 1.1 架构目标

| 目标 | 描述 | 优先级 |
|------|------|--------|
| **功能完整性** | 完整实现 D&D 5e 核心规则，支持完整的游戏流程 | P0 |
| **响应实时性** | 用户操作响应时间 < 100ms，LLM 响应流式输出 | P0 |
| **状态一致性** | Client 与 Server 状态强一致，无数据丢失 | P0 |
| **可扩展性** | 模块化设计，支持后续功能扩展（多用户、移动端） | P1 |
| **可维护性** | 清晰的分层架构，代码可读性好，便于调试 | P1 |
| **部署简便性** | 单进程部署，零外部依赖，开箱即用 | P1 |
| **资源效率** | 内存占用 < 500MB（不含前端），CPU 利用率合理 | P2 |

### 1.2 设计原则

| 原则 | 说明 | 实践方式 |
|------|------|----------|
| **分层架构** | 清晰的层次划分，每层只依赖下层 | API层 → 业务层 → 数据层 |
| **单一职责** | 每个模块只负责一个明确的功能 | Client/Server 模块分离 |
| **接口隔离** | 模块间通过明确定义的接口通信 | MCP 工具接口、WebSocket 消息协议 |
| **依赖倒置** | 高层模块不依赖低层实现细节 | LLM 适配器抽象接口 |
| **状态集中** | 游戏状态由 Server 统一管理 | 共享状态层 |
| **错误隔离** | 单个模块错误不影响整体系统 | 错误恢复机制、状态持久化 |

### 3.3 集成接口

#### 3.3.1 接口类型

| 接口类型 | 协议 | 用途 | 数据格式 |
|----------|------|------|----------|
| **实时通信** | WebSocket | 游戏操作、状态推送 | JSON |
| **管理接口** | REST API | 存档管理、配置查询 | JSON |
| **静态资源** | HTTP | 前端文件服务 | 静态文件 |

#### 3.3.2 接口边界

**前端负责**：
- 发送用户操作请求
- 接收并渲染状态更新
- 处理连接状态（重连）

**后端负责**：
- 验证请求合法性
- 执行业务逻辑
- 推送状态更新
- 处理异常情况

### 6.2 读写锁设计

```go
// internal/shared/state/manager.go

type StateManager struct {
    gameState   *GameState
    combatState *CombatState
    mu          sync.RWMutex
    combatMu    sync.RWMutex

    // 状态变更通知
    subscribers map[string]chan StateUpdate
    subMu       sync.RWMutex
}

// 读取状态（允许并发读）
func (sm *StateManager) GetGameState() *GameState {
    sm.mu.RLock()
    defer sm.mu.RUnlock()

    // 返回副本，避免外部修改
    return sm.gameState.Clone()
}

// 更新状态（独占写）
func (sm *StateManager) UpdateGameState(updater func(*GameState)) error {
    sm.mu.Lock()
    defer sm.mu.Unlock()

    // 执行更新
    updater(sm.gameState)

    // 通知订阅者
    sm.notifySubscribers(StateUpdate{
        Type: "game_state",
        Data: sm.gameState,
    })

    return nil
}

// 原子更新（事务性）
func (sm *StateManager) AtomicUpdate(fn func(*StateManager) error) error {
    sm.mu.Lock()
    defer sm.mu.Unlock()

    // 开始事务
    tx := sm.beginTransaction()
    defer tx.rollback() // 确保失败时回滚

    if err := fn(sm); err != nil {
        return err
    }

    tx.commit()
    return nil
}
```

### 6.3 状态更新原子性

```go
// 战斗行动原子更新示例
func (sm *StateManager) ExecuteCombatAction(action CombatAction) (*ActionResult, error) {
    sm.combatMu.Lock()
    defer sm.combatMu.Unlock()

    // 1. 验证
    if err := sm.validateAction(action); err != nil {
        return nil, err
    }

    // 2. 计算结果
    result := sm.calculateAction(action)

    // 3. 应用变更（原子操作）
    sm.applyActionResult(result)

    // 4. 持久化（可选异步）
    go sm.persistState()

    return result, nil
}

// 批量原子更新
func (sm *StateManager) BatchUpdate(updates []StateUpdate) error {
    sm.mu.Lock()
    defer sm.mu.Unlock()

    // 收集所有变更
    for _, update := range updates {
        if err := sm.applyUpdate(update); err != nil {
            // 回滚所有变更
            sm.rollbackUpdates()
            return err
        }
    }

    // 一次性通知
    sm.notifyBatchUpdate(updates)
    return nil
}
```

### 6.4 并发安全规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **读多写少用 RWMutex** | 允许多个读者，写者独占 | 状态查询 |
| **写操作必须 Lock** | 防止竞态条件 | 状态更新 |
| **避免嵌套锁** | 防止死锁 | 不同模块独立锁 |
| **锁内不调用外部** | 防止死锁 | 锁内不调用 I/O |
| **使用 defer 释放锁** | 确保锁释放 | `defer mu.Unlock()` |

### 6.5 死锁预防

```
锁获取顺序（必须严格遵守）：
1. StateManager.mu (全局状态)
2. StateManager.combatMu (战斗状态)
3. 持久化层锁

禁止：
- 反向获取锁
- 在持有锁时调用可能获取其他锁的函数
- 在锁内执行阻塞操作（网络、文件 I/O）
```

| 操作类型 | 锁类型 | 预期延迟 |
|----------|--------|----------|
| 状态查询 | RLock | < 1ms |
| 单字段更新 | Lock | < 5ms |
| 批量更新 | Lock | < 20ms |
| 战斗行动 | Lock | < 50ms |

## 8. 架构决策记录

### 8.1 关键决策

| 决策ID | 决策 | 选择 | 理由 |
|--------|------|------|------|
| A-001 | 部署方式 | 单进程合并部署 | 单用户场景，简化部署和运维 |
| A-002 | 前后端通信 | WebSocket | 双向实时通信需求，支持流式响应 |
| A-003 | 内部通信 | MCP 格式 | 统一的工具调用接口，便于 LLM 调用 |
| A-004 | 状态管理 | 后端集中式 | 单一真实来源，保证一致性 |
| A-005 | 状态存储 | 内存 + JSONL | 单用户本地场景，内存性能最优，JSONL 便于查看 |
| A-006 | 前端游戏引擎 | Phaser | 成熟的 2D 游戏框架，适合战棋 |
| A-007 | 后端框架 | Gin | 轻量高性能，Go 生态最成熟 |

### 8.2 架构风险

| 风险ID | 风险描述 | 缓解策略 |
|--------|----------|----------|
| AR-001 | LLM 响应延迟影响用户体验 | 流式响应 + 前端加载提示 |
| AR-002 | WebSocket 断线导致状态丢失 | 断线重连 + 状态持久化 |
| AR-003 | 前后端状态不一致 | 增量更新 + 定期全量同步 |
| AR-004 | 大量消息历史导致性能下降 | 历史压缩 + 分页加载 |
