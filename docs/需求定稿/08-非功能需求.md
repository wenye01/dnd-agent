# 非功能需求

> [返回索引](./index.md)

---

| 类别 | 需求描述 | 来源 |
| --- | --- | --- |
| **可扩展性** | 支持多 LLM 提供商（OpenAI、Anthropic） | 原始需求 |
| **可扩展性** | Web 前端采用低耦合架构，模块化设计 | 原始需求 |
| **可扩展性** | 后端支持多种前端接入（Web、移动端等） | 澄清 |
| **持久化** | 完整持久化：全部对话消息 + 游戏状态，支持多存档 | 澄清 |
| **互操作性** | Client-Server 内部通过 MCP 格式接口通信，合并部署 | 原始需求 |
| **实时性** | 数据变更即时同步，有修改即同步 | 澄清 |
| **可靠性** | 调用失败后使用持久化数据恢复 | 澄清 |
| **资源效率** | 上下文切换减少 token 消耗 | 原始需求 |
| **兼容性** | 地图采用 FVTT 格式（含墙壁、光照） | 澄清 |
| **规则严格性** | Server 严格执行 D&D 5e 规则，不允许规则外操作 | 澄清 |
| **用户体验** | Web 前端界面流畅，战斗动画平滑 | 推断 |
| **响应式设计** | Web 前端适配桌面和平板屏幕 | 推断 |
| **实时推送** | Server/Client 状态变化实时推送到 Web 前端 | 推断 |
| **技术栈** | Web 前端使用 React + Phaser | 澄清 |
| **通信协议** | Web 前端与游戏后端使用 WebSocket 通信 | 澄清 |
| **光照系统** | 对战斗探索无影响，保留数值供后续使用 | 澄清 |
| **FVTT 范围** | 只使用地图数据，不使用音频、宏等功能 | 澄清 |
| **可靠性** | LLM 调用失败重试机制：最多重试 3 次，指数退避间隔（1s, 2s, 4s） | 技术需求 |
| **一致性** | 状态同步采用乐观锁机制，失败时回滚到上一致状态 | 技术需求 |
