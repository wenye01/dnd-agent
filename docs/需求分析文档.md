# 需求分析文档 v7

> **版本说明**：本文档基于 v6 更新，明确了术语定义（游戏后端）、简化上下文系统设计、明确了暂不实现和待后续设计的内容。

## 1. 项目概述

### 1.1 项目背景

这是一个基于 D&D 5e 规则的 AI 驱动角色扮演游戏系统。核心思想是让 LLM（大语言模型）扮演 DM（地下城主）角色，用户通过自然语言与系统交互，体验完整的 D&D 游戏流程。

**参考游戏**：博德之门 3（Baldur's Gate 3）—— 战斗、地图交互、战棋模式参考

### 1.2 涉及方/角色

| 角色         | 类型        | 描述                          |
| ---------- | --------- | --------------------------- |
| **用户**     | 人类玩家      | 使用自然语言与 Web 前端交互，控制一个或多个角色 |
| **玩家角色**   | 游戏实体      | 由用户控制的角色（可单角色或多角色队伍）        |
| **AI 角色**  | 游戏实体      | NPC 和敌人，由 AI 控制行为           |
| **游戏后端（Backend）** | 后端服务     | **Client 模块与 Server 模块的整合**：接收用户输入，管理 LLM 会话和上下文，执行工具调用，运行 D&D 5e 规则引擎，使用go语言 |
| **LLM**    | 远程 API 服务 | 充当 DM 角色，控制 NPC/敌人，推进游戏     |
| **Web 前端** | Web 应用    | **核心交互界面**：地图展示、用户输入、战斗可视化、状态监控、管理操作 |

### 1.3 项目范围

**包含**：

- 完整的 D&D 5e 游戏流程（角色创建、剧本选择、探险、战斗）
- 多 LLM 提供商支持（OpenAI、Anthropic API 格式）
- MCP 协议通信
- 会话管理和上下文切换
- **Web 前端界面**：
  - 地图系统（冒险地图 + 场景地图，FVTT 格式）
  - 用户输入界面（自然语言 + 地图交互）
  - 战斗过程可视化（战棋模式，参考博德之门 3）
  - Server 状态监控
  - 管理操作界面
- 战棋式战斗系统（参考博德之门 3）
- **多存档系统**：支持多个存档，存档命名（可选）

**当前版本不包含**：

- 多用户联机（后续版本考虑）
- 用户自定义剧本创建
- 移动端支持（后续版本考虑，但后端已支持多前端接入）

**后续版本计划**：

- 多用户支持
- 移动端界面
- 光照系统对战斗/探索的影响

**数据流向说明**：

1. **用户 → Web 前端**：用户通过 Web 界面进行输入（自然语言/地图点击/战斗操作）
2. **Web 前端 → 游戏后端**：通过 **WebSocket** 将用户操作传递给 Client 模块处理
3. **Client → LLM**：Client 模块将用户输入发送给 LLM 进行理解和决策
4. **LLM → Client**：LLM 返回叙述内容或工具调用指令
5. **Client → Server**：通过 MCP 格式接口调用 Server 模块工具（内部调用，无网络开销）
6. **Server → Client**：返回执行结果，更新共享游戏状态
7. **游戏后端 → Web 前端**：通过 **WebSocket** 推送状态更新、战斗事件、执行过程到 Web 前端展示
8. **Web 前端 → 用户**：可视化展示（React UI + Phaser Canvas 渲染地图、战斗动画、叙述文本等）

### 3.3 核心业务流程

#### 3.3.1 创建游戏流程

```
1. 用户输入开始游戏的自然语言描述
2. Client 接收文本，调用 LLM
3. LLM 理解用户需求，调用"创建会话"工具
4. Client 解析工具调用，请求 Server 创建会话
5. Server 创建会话，返回 sessionId 及其他信息
6. Client 使用 sessionId 创建本地会话
7. 实时同步 Client-Server 游戏状态
8. 创建完成
```

#### 3.3.2 创建角色流程

```
1. 游戏创建成功后，LLM 引导用户创建角色
2. 用户回复角色创建信息（种族、职业、属性等）
3. Client 传递给 LLM
4. LLM 调用 Server 工具创建角色
5. Server 按 D&D 5e 规则验证并创建角色
6. 返回成功/错误信息（错误时提示缺少/错误内容）
7. 可重复创建多个角色（队伍模式）
8. 实时同步游戏状态
```

#### 3.3.3 选择剧本流程

```
1. Web 前端呈现可选剧本列表
2. 用户在 Web 前端选择剧本
3. Web 前端将选择发送给 Client 模块
4. Client 接收选择，发送给 LLM
5. LLM 调用 Client 的加载剧本功能
6. Client 加载剧本，必要信息载入上下文
```

#### 3.3.4 游戏过程流程

```
循环:
  1. LLM 作为 DM 叙述背景/过程，或控制 NPC 行动
  2. 用户做出行动（或 LLM 控制的敌人行动）
  3. Client 接收后发给 LLM
  4. LLM 根据输入选择:
     - 继续叙述
     - 回答用户问题
     - 调用 Server 工具（检定、技能、交互等）
  5. Server 严格执行 D&D 5e 规则，不允许规则外操作
  6. 如果进入战斗:
     a. 自动切换到战斗上下文
     b. 进入战棋模式（参考博德之门 3）
     c. 按先攻顺序循环
     d. 战斗结束后恢复上下文
  7. 实时同步所有状态变化
```

#### 3.3.5 地图切换流程

```
1. AI DM 决定进入场景
2. 根据场景标签加载对应的场景地图
3. 进入场景地图，角色放置到入口位置
4. 在场景地图中移动、交互
5. 离开场景返回冒险地图
```

### 3.4 功能详情

#### 3.4.1 上下文系统

上下文是 LLM 进行决策和叙述时所能访问的全部信息。系统需要有效管理上下文，在保证 LLM 有足够信息的同时控制 token 消耗。

##### 3.4.1.1 上下文内容分类

上下文内容分为以下几大类：

| 类别 | 特性 | 描述 |
| --- | --- | --- |
| **剧本数据** | 静态 | 从 PDF 剧本解析得到的固定内容 |
| **游戏运行时状态** | 动态 | 随游戏进行不断变化的状态 |
| **历史会话** | 累积 | 对话记录、行动历史、事件记录 |
| **规则参考** | 静态 | D&D 5e 规则摘要、剧本特定规则 |
| **用户设置** | 持久 | 用户偏好、显示设置、快捷键等 |

> **说明**：各类别的具体数据结构将在技术设计阶段详细定义。

##### 3.4.1.2 上下文切换机制

- **目的**：管理 token 消耗，优化 LLM 性能
- **触发条件**：
  - **自动触发**：进入战斗时自动切换到战斗上下文
  - **LLM 决定**：其他情况由 LLM 决定是否切换
- **上下文类型**：
  - **探索上下文**：默认状态，包含角色卡、位置、章节目标等
  - **战斗上下文**：进入战斗时自动切换，包含战斗属性、敌人数据、战斗规则等
  - **对话上下文**：与重要NPC深度对话时，包含NPC完整数据、对话历史等

##### 3.4.1.3 历史会话压缩

- **目的**：控制上下文窗口大小
- **策略**：预留压缩接口，具体实现待后续阶段设计

#### 3.4.2 战斗系统（战棋模式）

- **参考**：博德之门 3
- **核心特点**：
  - 战斗在场景地图上进行
  - 支持角色移动、距离计算
  - 掩护判定、机会攻击
  - 墙壁、光照影响战斗
- **规则执行**：
  - Server 严格执行 D&D 5e 规则
  - **不允许**执行规则外的操作
- **调用方式**：
  1. 细粒度：LLM 依次调用检定、掷骰子等
  2. 高层抽象：LLM 调用攻击，Server 自动完成全部流程

#### 3.4.3 实时数据同步

- **机制**：有修改即同步（非传统心跳）
- **同步内容**：全部游戏状态
- **恢复**：调用失败时可以进行重试，等待重新连接，退出后使用持久化数据恢复

#### 3.4.4 地图系统

| 地图类型     | 用途    | 特点                       |
| -------- | ----- | ------------------------ |
| **冒险地图（WorldMap）** | 世界大地图 | 显示大致范围、玩家当前位置、各地点信息      |
| **场景地图（SceneMap）** | 具体场景  | FVTT 格式，支持移动、交互、遮挡、墙壁、光照 |

**场景地图特性**：

- 标签系统：不同地点可进入不同场景
- FVTT 数据：墙壁、光照、网格
- 交互元素：参考博德之门 3 的 RPG 交互

**FVTT 使用范围**：

> **说明**：FVTT 地图数据映射按照官方格式来，直接设计地图系统，和现有系统能够一起工作即可。

| 功能 | 是否使用 | 说明 |
| --- | --- | --- |
| **地图数据（墙壁、光照、网格）** | ✅ 使用 | 核心地图数据 |
| **音频系统** | ❌ 不使用 | 当前版本不使用 |
| **宏系统** | ❌ 不使用 | 当前版本不使用 |
| **Token 系统** | 部分使用 | 参考数据结构，自行实现角色显示 |
| **光源数据** | ✅ 使用 | 保留数值，对战斗探索无影响，供后续版本使用 |

### 5.2 系统能力定义

> **说明**：本节描述各系统组件应该具备的能力，而非具体接口定义。能力定义关注"能做什么"，接口设计将在后续架构设计阶段确定。

#### 5.2.1 Server 模块应对外提供的能力

Server 模块作为 D&D 5e 规则引擎，通过 MCP 格式接口暴露以下能力：

| 能力领域     | 能力描述                                                                 |
| -------- | -------------------------------------------------------------------- |
| **会话管理** | 能够创建、查询、同步游戏会话；能够返回会话当前状态及缺失信息                                       |
| **角色管理** | 能够按 D&D 5e 规则创建、更新、删除角色；能够验证角色数据的合法性                                  |
| **剧本管理** | 能够加载内置剧本；能够返回剧本相关信息供 Client 使用                                       |
| **检定能力** | 能够执行各种检定（属性检定、技能检定、豁免检定）；能够处理优势/劣势；能够掷骰子并返回结果                        |
| **战斗能力** | 能够发起、进行、结束战斗；能够处理先攻顺序；能够执行攻击、施法、移动等战斗动作；能够按 D&D 5e 规则计算伤害、状态效果        |
| **地图能力** | 能够加载、切换地图；能够处理角色在地图上的移动；能够计算距离、掩护、视线等；能够处理与地图元素的交互                   |
| **休息恢复** | 能够处理短休息（HD 消耗恢复 HP）；能够处理长休息（全面恢复）                                    |
| **物品能力** | 能够管理角色背包；能够处理物品的获取、使用、装备、丢弃                                          |
| **法术能力** | 能够管理法术位；能够验证施法条件；能够执行法术效果                                           |
| **状态同步** | 能够将游戏状态变更实时同步给 Client；能够在 Client 请求时返回最新状态                            |
| **错误处理** | 能够返回清晰的错误信息，说明无法执行的原因（如违反规则、缺少前置条件等）                                 |

#### 5.2.2 Client 与 Server 模块的协作

Client 和 Server 模块在同一进程内运行，通过 MCP 格式接口通信，共享游戏状态和会话数据：

| 能力领域     | 能力描述                                                                 |
| -------- | -------------------------------------------------------------------- |
| **身份标识** | 能够提供会话标识，让 Server 知道是哪个 Client/会话在进行请求                               |
| **状态接收** | 能够接收 Server 推送的游戏状态更新                                                |
| **结果处理** | 能够接收 Server 返回的操作结果（成功/失败、详细数据）                                      |

#### 5.2.3 Client 内部应具备的能力

Client 作为 AI Agent，内部需要具备以下能力：

| 能力领域      | 能力描述                                                              |
| --------- | ----------------------------------------------------------------- |
| **LLM 通信** | 能够调用 OpenAI/Anthropic 格式的 LLM API；能够处理流式响应                        |
| **工具调用**  | 能够解析 LLM 返回的工具调用指令；能够执行工具调用并将结果返回给 LLM                             |
| **会话管理**  | 能够管理多个会话；能够持久化会话（对话消息 + 游戏状态）；能够从持久化数据恢复会话                         |
| **上下文管理** | 能够构建和维护 LLM 上下文；能够根据场景（如进入战斗）自动切换上下文；能够控制上下文大小以优化 token 消耗         |
| **剧本加载**  | 能够加载剧本信息；能够将必要信息注入到 LLM 上下文中                                      |
| **用户交互**  | 能够接收用户自然语言输入；能够将 LLM 响应呈现给用户                                      |
| **MCP 通信** | 能够通过 MCP 协议与 Server 通信；能够发现和调用 Server 提供的工具                        |
| **状态同步**  | 能够与 Server 保持游戏状态同步；能够在连接断开后使用持久化数据恢复                              |
| **错误处理**  | 能够处理 LLM 调用失败；能够处理 Server 调用失败；能够向用户呈现友好的错误信息                     |

#### 5.2.4 Web 前端系统应具备的能力

Web 前端作为用户交互的核心界面，采用 **React + Phaser** 双层架构，需要具备以下能力：

| 能力领域       | 技术层 | 能力描述                                                                 |
| ---------- | ---- | -------------------------------------------------------------------- |
| **地图渲染**   | Phaser | 能够渲染冒险地图（世界概览）和场景地图（FVTT 格式）；能够显示墙壁、光照、交互元素                          |
| **角色显示**   | Phaser | 能够在地图上显示角色位置和状态；能够显示角色移动动画                                           |
| **战斗可视化**  | Phaser | 能够显示战棋网格；能够高亮当前行动角色；能够显示移动范围和攻击范围；能够播放攻击/法术动画；能够显示伤害数字和状态效果          |
| **用户输入**   | React | 能够接收自然语言文本输入；能够处理地图点击交互；能够提供快捷操作按钮                                   |
| **对话展示**   | React | 能够展示 DM 叙述和对话历史；能够区分不同说话者；能够支持滚动查看历史                                 |
| **角色面板**   | React | 能够展示角色属性、HP、AC、状态效果；能够展示背包物品；能够展示已知/准备法术                             |
| **状态监控**   | React | 能够实时展示 Server 内部状态（游戏进度、角色状态等）；能够展示 Server 执行过程（检定 → 掷骰子 → 执行）        |
| **管理操作**   | React | 能够执行管理操作（强制结束战斗、修改数据等）                                               |
| **存档管理**   | React | 能够管理多个存档；支持创建、加载、删除存档；存档命名（可选）                                    |
| **WebSocket通信** | 共用 | 能够通过 WebSocket 与游戏后端建立实时双向连接；能够处理断线重连                                       |
| **实时更新**   | 共用 | 能够接收并响应 Server/Client 推送的状态更新；能够实时反映游戏状态变化                           |
| **响应式布局** | React | 能够适配不同屏幕尺寸（桌面/平板）                                                    |

### 5.3 交互能力约定

#### 5.3.1 Web 前端 ↔ Client/Server 交互约定

| 约定项        | 说明                                                                     |
| ---------- | ---------------------------------------------------------------------- |
| **通信协议**   | 使用 WebSocket 进行实时双向通信                                                 |
| **输入传递**   | Web 前端接收用户输入后，通过 WebSocket 传递给 Client 进行处理                             |
| **状态订阅**   | Web 前端订阅 Server/Client 的状态更新事件                                         |
| **实时推送**   | Server/Client 状态变化时，应能通过 WebSocket 实时推送到 Web 前端                        |
| **地图数据**   | Server 应能向 Web 前端提供地图数据（FVTT 格式）                                       |
| **战斗事件**   | 战斗中的行动、伤害、状态变化等事件应能推送到 Web 前端进行可视化                                      |
| **多前端支持** | 后端应设计为支持多种前端接入（Web、移动端等），通信协议统一为 WebSocket                              |

#### 5.3.3 Client ↔ Server 交互约定

| 约定项        | 说明                                                                     |
| ---------- | ---------------------------------------------------------------------- |
| **部署方式**   | Client 与 Server 合并部署在同一进程，内部解耦                                          |
| **接口格式**   | 内部调用遵循 MCP 接口格式（工具定义：name, description, inputSchema），但无网络/序列化开销        |
| **数据共享**   | 游戏状态、会话历史等数据在进程内共享，Client 和 Server 模块可直接访问                              |
| **状态权威**   | Server 模块为规则权威，负责状态的计算和验证                                              |
| **错误反馈**   | Server 应能返回清晰的错误信息，Client 应能将错误信息传递给 LLM 处理                             |
| **渐进式披露** | Client 向 LLM 披露工具信息采用三层渐进式：① 始终提供工具索引（名称+简述）；② 进入对应场景时向 LLM 提供该场景工具的详细定义；③ 调用时按需提供示例和参考文档。Server 端工具始终可用，披露控制发生在 Client 侧的上下文管理层 |

#### 5.3.4 Client ↔ LLM 交互约定

| 约定项      | 说明                                                                     |
| -------- | ---------------------------------------------------------------------- |
| **工具发现** | Client 应能将 Server 提供的工具列表注入到 LLM 上下文中                                  |
| **工具执行** | LLM 决定调用工具时，Client 应能解析并执行，将结果返回                                       |
| **上下文控制** | Client 应能根据场景控制上下文内容，优化 token 使用                                       |

| **战棋模式** | 回合制战斗，角色在地图上移动，参考博德之门 3 |

| **剧本数据** | 从 PDF 剧本解析得到的静态只读数据 |
| **运行时状态** | 随游戏进行不断变化的动态数据 |
| **渐进式披露** | 按三层结构向 LLM 披露工具：索引 → 激活 → 执行 |
| **检定** | D&D 中通过掷骰子来确定某项行动是否成功的机制 |
| **DC** | Difficulty Class，难度等级 |
| **优势/劣势** | 两次掷骰取高/低 |
| **HD** | Hit Dice，生命骰，用于短休息恢复 HP |
| **AC** | Armor Class，护甲等级 |
| **CR** | Challenge Rating，挑战等级 |
| **遭遇** | 剧本中设计的战斗、社交、解谜等事件 |
| **事件触发器** | 满足特定条件时自动触发的游戏事件 |
| **实时同步** | 数据变更时立即同步，非定时心跳 |
| **实时推送** | Server/Client 状态变化主动推送到 Web 前端 |

### 9.1 原始需求

```
整体规划

用户，client，llm，server

用户：游玩产品的自然人，只会使用自然语言，输入到client里

client：接受用户输入，传递给llm，能够解析llm传过来的工具调用，然后执行或请求server执行，管理整个会话以及上下文，内部要能支持多个llm，要能够支持openai，anthropic api两种格式的llm调用，类似于一个正常的ai agent，使用内存存储，持久化用jsonl文件

llm：远端llm api服务

server：dnd游戏引擎，能够自动运行游戏过程，包括战斗，状态管理，背包，角色等等dnd5e规则有的内容，

client和server合并部署在同一进程，内部解耦，通过MCP格式接口通信，共享游戏状态和会话数据

程序启动

client加载server的tool列表进去，就像一个正常Agent加载mcp一样

创建游戏：

用户输出开始游戏的自然语言描述 --> client接受到文本，调用LLM -->llm理解了用户需要，调用工具创建会话--> client接受到工具调用指令，解析参数，调用到了server的创建会话工具-->server内部创建会话，然后返回相应的sessionid以及其他信息给client-->client使用这个id创建会话，与server端同步。创建游戏完成

（server应该能提供一个查询游戏状态的接口，如果缺少一些内容没有创建，client调用的时候要返回缺少什么内容，然后llm请求用户补充）

创建角色：

接上述，创建完游戏后，时序在client这里

client将成功创建的信息传递给LLM-->llm了解到创建成功，开始游戏的下一流程，要求玩家创建角色-->用户回复角色创建内容-->client接受到后交给llm-->llm调用server工具创建角色-->server创建角色，根据之前提到的，应该返回角色创建成功，或者错误信息，

client和server间应该有一个心跳通信，同步游戏状态，每一次llm调用，游戏状态都应该在里面，让llm能够得知是否可以开始游戏

创建剧本：

server内会有内置剧本，用户可以选择

选择剧本时：

client给用户呈现可选剧本-->用户选择剧本-->client接收到发送给llm-->llm使用工具调用client的加载剧本功能-->client加载剧本，必要信息载入上下文

游戏过程：

LLM作为DM，开始叙述背景，过程等内容-->用户对此做出行动-->client接收到发给llm-->

llm根据用户文本，选择继续叙述/回答用户问题/调用server工具，

然后循环

进入战斗后按照dnd5e战斗规则循环，server需要提供检定，掷骰子等功能，这些功能由llm调用，此外llm也可以直接调用攻击，server在内部自动进行检定，掷骰子，然后返回结果

其余过程类似，此外client需要能够切换上下文，即进入战斗后，切换到战斗使用的上下文，把如故事背景等一些战斗关系不大，时间离的远的内容给去掉，切换到专用于战斗的上下文。

server需要支持类似控制台的功能，即可以有一个web实时监控状态，并且把内部执行过程暴露出来，比如client调用server的攻击，web端可以看到server检定，掷骰子，执行攻击这样的过程，就像一个rpg游戏一样，这个功能架构上不要太耦合，最好工具注册个什么类就能完成

client和server合并部署在同一进程，内部解耦，通过MCP格式接口通信，共享游戏状态和会话数据，server需要严格按照dnd5e来设计并提供功能

地图需求

地图分两块，冒险地图，即大的世界地图，以及场景地图，冒险地图只提供大致范围，玩家当前在哪，某地有什么这种，
```

### 9.2 需求澄清记录

```
8.1 系统设计：
- 暂时只支持单用户使用，多用户后续考虑
- 会话恢复按持久化内容，包含全部对话消息和游戏状态，不一定全用 JSONL
- 用户可控制单角色或多角色，NPC 和敌人由 AI 控制

8.2 剧本系统：
- PDF 需要先解析成格式化数据
- 剧本包含地点、NPC、事件触发条件、战斗遭遇等全部内容
- 用户目前不可自定义剧本

8.3 游戏机制：
- NPC 和敌人由 AI 控制，像用户操作的角色一样
- 不允许执行规则外的事情
- 战斗需要地图位置信息，像博德之门 3 一样的战棋游戏

8.4 地图系统：
- 冒险地图某点可进入场景地图，或由 AI DM 决定进入
- 场景地图有标签，不同地方进入的地图不同
- 交互像 RPG 游戏，参考博德之门
- 战斗在场景地图进行，参照博德之门
- 使用 FVTT 地图数据，包括墙壁、灯光

8.5 技术实现：
- "心跳"是实时同步，有修改就同步
- 上下文切换：LLM 决定，但进入战斗自动切换
- 调用失败用持久化数据恢复
- Web 端可进行管理操作

额外要求：
- 按 D&D 5e 规则查缺补漏，5e 规则有的内容都应有

8.6 Web 前端定位（v4 更新）：
- Web 端不仅仅是监控 Server，而是整个前端界面
- 地图展示：冒险地图、场景地图都在 Web 端显示
- 用户输入：自然语言输入、地图点击交互、快捷操作按钮
- 战斗过程：战棋网格、角色移动、攻击动画、伤害数字、状态效果可视化
- 信息展示：角色面板、对话历史、剧本信息
- 保留监控与管理功能

8.7 需求明确（v5 更新）：

1. **上下文切换**：需要首先明确有哪些内容，然后才能确认。
   - 建议上下文分类：探索、战斗、对话、休息四种
   - 进入战斗时自动切换到战斗上下文

2. **核心数据实体**：
   - 角色卡（Character）：玩家角色数据
   - 生物卡（Creature）：NPC、敌人等所有生物的统一数据结构
   - 物品（Item）：游戏物品
   - 冒险背景（AdventureBackground）：世界观和背景故事
   - 剧情（Chapter）：分章节的剧情内容
   - NPC：非玩家角色
   - 地图：分大地图（WorldMap）和场景地图（SceneMap）

3. **存档系统**：
   - 支持多存档
   - 存档命名支持，但命名非必要（可选）

4. **光照系统**：
   - 对战斗、探索无实际影响
   - 保留数值，后续版本可能使用

5. **FVTT 使用范围**：
   - 目前只使用地图数据（墙壁、光照、网格）
   - 其余功能（音频、宏等）不使用

6. **Web 前端技术栈**：
   - React 为主，做 UI 界面
   - Phaser 为 Canvas 嵌入，做战斗场景、地图等更"游戏"的内容

7. **通信方式**：
   - 使用 WebSocket 进行 Web 前端与游戏后端之间的实时通信

8. **移动端支持**：
   - 当前不需要移动端
   - 后续可能添加
   - 后端应该能够支持多种前端接入

9. **场景划分**：
   - 先给出当前可能拥有的场景，然后决策
   - 建议场景：探索、战斗、对话、休息（与上下文分类对应）

8.8 需求明确（v6 更新）：

1. **上下文内容分类**：
   - 明确五大类：剧本数据（静态）、游戏运行时状态（动态）、历史会话、规则参考、用户设置
   - 剧本数据：从 PDF 解析得到的只读数据
   - 运行时状态：随游戏变化的动态数据

2. **剧本解析数据结构**（基于《风骸岛之龙》PDF 分析）：
   - 元信息：剧本ID、名称、版本、推荐等级等
   - 冒险概述：钩子、背景故事、概要、势力关系
   - 章节：章节ID、目标、事件、完成条件
   - 地点：位置、描述、交互元素、NPC引用、遭遇引用
   - NPC：属性、态度、对话脚本、任务关联
   - 生物：属性、能力、行为模式
   - 遭遇：类型、触发条件、敌人配置、奖励
   - 物品：属性、效果、位置
   - 事件：触发条件、触发效果
   - 世界地图：区域、地点标记
   - 附录：随机表、朗读文本

3. **游戏运行时状态**：
   - 会话信息：sessionId、存档信息、游戏时长
   - 进度状态：当前章节、已完成目标、已触发事件、已访问地点
   - 当前场景：位置、角色状态、NPC状态、环境状态
   - 战斗状态：回合、先攻顺序、战斗参与者状态
   - 角色状态：HP、法术位、状态效果、背包、装备
   - 世界状态：势力关系变化、NPC状态变化、全局标记

4. **历史会话结构**：
   - 对话消息：角色、内容、时间戳、关联动作
   - 行动记录：执行者、类型、详情、结果
   - 掷骰记录：类型、骰子、结果、修正值
   - 重要决策：描述、选择、影响

5. **上下文类型扩展**：
   - 从四种扩展到六种：探索、战斗、对话、休息、购物、解谜
   - 每种类型定义核心内容和辅助内容
```
